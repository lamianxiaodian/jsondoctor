JSON in REST APIs: Best Practices

JSON has become the standard format for REST APIs due to its simplicity and wide language support. This guide covers best practices for designing and implementing JSON-based REST APIs.

Request/Response Structure

Standard Request Headers

POST /api/users HTTP/1.1
Content-Type: application/json
Accept: application/json
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...

{
  "name": "John Doe",
  "email": "john@example.com"
}

Standard Response Headers

HTTP/1.1 201 Created
Content-Type: application/json
Location: /api/users/123

{
  "id": 123,
  "name": "John Doe",
  "email": "john@example.com",
  "createdAt": "2024-01-15T10:30:00Z"
}

Consistent Response Format

Success Response

{
  "success": true,
  "data": {
    "id": 123,
    "name": "John Doe"
  },
  "meta": {
    "timestamp": "2024-01-15T10:30:00Z"
  }
}

Error Response

{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Invalid input data",
    "details": [
      {
        "field": "email",
        "message": "Invalid email format"
      }
    ]
  }
}

Pagination

{
  "success": true,
  "data": [...],
  "pagination": {
    "page": 1,
    "perPage": 20,
    "totalPages": 5,
    "totalItems": 98
  },
  "links": {
    "first": "/api/users?page=1",
    "prev": null,
    "next": "/api/users?page=2",
    "last": "/api/users?page=5"
  }
}

Naming Conventions

Use camelCase for JSON Keys

// Good
{
  "firstName": "John",
  "lastName": "Doe",
  "createdAt": "2024-01-15T10:30:00Z"
}

// Avoid
{
  "first_name": "John",
  "FirstName": "John",
  "first-name": "John"
}

Handling Dates and Times

Always use ISO 8601 format for dates and times in JSON APIs.

{
  "createdAt": "2024-01-15T10:30:00Z",
  "updatedAt": "2024-01-15T10:30:00.123Z",
  "birthDate": "1990-05-20"
}

Handling Null vs. Omitted Fields

// Include null for explicitly null values
{
  "name": "John",
  "middleName": null,
  "lastName": "Doe"
}

// Omit optional fields that weren't provided
{
  "name": "John",
  "lastName": "Doe"
}

HTTP Status Codes with JSON

  • 200 OK: Successful GET, PUT, PATCH
  • 201 Created: Successful POST that creates a resource
  • 204 No Content: Successful DELETE
  • 400 Bad Request: Invalid JSON or validation errors
  • 401 Unauthorized: Authentication required
  • 403 Forbidden: Authenticated but not authorized
  • 404 Not Found: Resource doesn't exist
  • 422 Unprocessable Entity: Validation errors
  • 500 Internal Server Error: Server error

Versioning

// URL versioning
GET /api/v1/users
GET /api/v2/users

// Header versioning
Accept: application/vnd.myapi.v1+json
Accept: application/vnd.myapi.v2+json

Filtering, Sorting, and Searching

// Filtering
GET /api/users?status=active&role=admin

// Sorting
GET /api/users?sort=-createdAt,name

// Searching
GET /api/users?search=john

// Field selection
GET /api/users?fields=id,name,email

Batch Operations

POST /api/users/batch
{
  "operations": [
    {
      "action": "create",
      "data": {"name": "User 1"}
    },
    {
      "action": "update",
      "id": 123,
      "data": {"name": "Updated Name"}
    },
    {
      "action": "delete",
      "id": 456
    }
  ]
}

Best Practices Summary

  • Use consistent naming conventions (camelCase)
  • Return appropriate HTTP status codes
  • Include meaningful error messages
  • Use ISO 8601 for dates
  • Implement pagination for collections
  • Version your API
  • Document your JSON schemas
  • Validate all incoming JSON
